**무엇을**: `thumbnail_view()`가 `st.components.v1.html(...)`(iframe `srcdoc`)로 위젯을 렌더하던 것을, `st.components.v1.declare_component("coupang_thumbnail_widget", path=component_thumbnail/)`로 교체. 위젯 HTML은 이제 `component_thumbnail/index.html`로 매 렌더마다 (현재 EXTENSION_ID로 치환해) 저장되고, Streamlit 자체 서버가 실제 경로(`/component/<name>/index.html`)로 서빙한다. 위젯 JS에 `streamlit:componentReady`/`streamlit:setFrameHeight` postMessage 핸드셰이크를 최소 구현으로 추가(프레임 높이 자동조정용, Streamlit 커스텀 컴포넌트 프로토콜).

**영향**: `component_thumbnail/`은 런타임에 자동 생성·재작성되는 빌드 산출물이라 `.gitignore`에 추가(커밋 안 함). `declare_component`는 실제 브라우저 세션(WebSocket 스크립트 실행) 안에서만 컴포넌트가 레지스트리에 등록되므로, 정적 curl/헬스체크만으로는 서빙 여부를 끝까지 검증할 수 없었다 — Python 로직(파일 생성·치환)과 Chrome 공식 매치패턴 규칙까지만 독립적으로 확인했고, 실제 "웹앱에서 확장이 응답하는지"는 사용자가 브라우저에서 최종 확인해야 한다.
**되돌리기**: 필요 시 `thumbnail_view()`를 이전 커밋(`components.html` 사용 버전)으로 되돌릴 수 있으나, 되돌리면 이 버그가 재발한다(권장 안 함).

## [2026-07-02] 백그라운드(비활성) 탭 → 활성 탭으로 변경 (봇 차단 우회 시도)
**무엇을**: `extractFromUrl()`가 쿠팡 페이지를 열 때 `chrome.tabs.create({url, active:false})` → `active:true`로 변경. 추출 후 탭을 닫으면서 원래 웹앱 탭(`sender.tab.id`)으로 자동 포커스 복귀. SPA/봇체크 대기시간도 1200ms→2500ms로 늘림.

**영향**: 확장 사용 시 쿠팡 상품 탭이 화면에 잠깐(1~2초) 나타났다 자동으로 닫히고 원래 화면으로 돌아옴 — 완전히 안 보이던 이전 방식보다 화면이 살짝 깜빡이지만, 봇 차단 우회가 우선순위이므로 감수. 이것도 100% 해결 보장은 아님(사용자 실측 필요) — 만약 이래도 안 되면 다음 후보는 탭을 더 오래 유지하거나, 확장의 popup.js(사용자가 쿠팡 탭에서 직접 클릭)로 완전히 전환하는 것.

**[정정, 같은 날]** 사용자가 "필터 개선을 요청하기 전에는 탭이 눈에 안 보이는 채로도(active:false) 잘 됐었다"고 확인 → "비활성 탭이 봇차단 원인"이라는 가설이 근거 부족으로 판명되어 `active:false`로 되돌림. **재테스트 결과: 원래 성공했던 링크 + 다른 새 링크 모두 정상 동작 확인.** 즉 그 "0개" 증상은 `active:false` 자체의 문제가 아니었고, 같은 세션에서 반복 테스트하며 발생한 일시적 현상(쿠팡 측 rate-limit 의심)이거나 하이드레이션 대기시간을 1200→2500ms로 늘린 게 실제 도움이 됐을 가능성이 있다 — 정확한 원인은 미확정이지만 현재 설정(active:false, 대기 2500ms)으로 재현 안 됨을 확인. 재발 시 debug 필드로 재진단.

## [2026-07-02] 라이센스(로그인) 시스템 전면 제거 → URL 쿼리파라미터 매니저 구분
**무엇을**: `login_view()`/`license_view()`/`ROLE_KO`·`ROLE_EN`/`storage.py`의 라이센스 CRUD(`get_license`,`list_licenses`,`add_license`,`delete_license`) 전부 삭제. `schema.sql`에서 `cp_licenses` 테이블 생성 제거(기존에 이미 만든 Supabase의 `cp_licenses` 테이블은 건드리지 않음 — 앱이 더는 안 읽을 뿐, DROP은 안 함). 대신 `_is_manager()`가 URL 쿼리파라미터 `?admin=<MANAGER_TOKEN>` 을 서버 설정값(`MANAGER_TOKEN`, env/secrets)과 비교해 매니저 여부를 판별. 사이드바는 "매니저 모드"/"팀원 모드"만 표시(개인 식별 없음).
**이유**: 사용자가 "새로고침하면 로그인(라이센스)이 초기화돼 매우 불편하다"고 지적. 원인은 Streamlit이 새로고침 시 세션을 새로 만들어 `st.session_state`가 초기화되는 프레임워크 기본 동작. 세션이 아니라 URL 자체에 상태를 두면 새로고침에도 유지되므로, 사용자가 직접 "라이센스를 아예 없애자"로 방향을 틀었고, "한 URL로 팀원/매니저 구분 가능한지" 확인 후(가능 — 쿼리파라미터로) 이 방식으로 확정.
**영향**: 팀원별 개인 식별(누가 언제 썼는지)은 사라짐. 팀원용 링크변환·썸네일 기능은 URL만 알면 로그인 없이 완전히 열려 있음(원래도 사실상 그랬음). 매니저 토큰이 담긴 URL이 유출되면 그 사람도 API키 관리 가능 — URL 공유 시 주의 필요. 로컬 테스트용 `MANAGER_TOKEN`은 이 프로젝트 `.env`에 무작위로 생성해 넣어둠(값은 채팅에 남기지 않음, 전역 안티패턴 규칙 준수). 배포 시 Streamlit Secrets에도 동일하게 넣어야 함.
**되돌리기**: git에서 이전 커밋(로그인 화면 버전)으로 `app.py`/`storage.py`/`schema.sql`을 되돌리면 됨. 단, 그 경우 다시 "새로고침 시 로그인 풀림" 문제가 재발함(구조적 한계).
